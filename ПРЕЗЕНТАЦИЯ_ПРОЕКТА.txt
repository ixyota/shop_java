1. ОБЩЕЕ ОПИСАНИЕ ПРОЕКТА
───────────────────────────────────────────────────────────────────────────────

Проект представляет собой полнофункциональный веб-сайт онлайн-магазина с 
современным дизайном и административной панелью управления.

Основные возможности:
  • Просмотр товаров и категорий
  • Поиск товаров
  • Фильтрация по категориям
  • Корзина покупок
  • Административная панель для управления товарами и категориями
  • Загрузка изображений для товаров
  • Адаптивный дизайн для всех устройств


2. ТЕХНОЛОГИЧЕСКИЙ СТЕК
───────────────────────────────────────────────────────────────────────────────

ФРОНТЕНД:
  • HTML5 - структура страниц
  • CSS3 - современный адаптивный дизайн с градиентами и анимациями
  • JavaScript (ES6+) - интерактивность, работа с API, управление состоянием
  • Адаптивная верстка (mobile-first подход)

БЭКЕНД:
  • Java 17 - основной язык программирования
  • Spring Boot 3.2.0 - фреймворк для создания веб-приложений
  • Spring Data JPA - работа с базой данных через ORM
  • REST API - архитектура для взаимодействия фронтенда и бэкенда

БАЗА ДАННЫХ:
  • H2 Database (файловая) - встроенная база данных
  • JPA/Hibernate - ORM для работы с БД
  • Автоматическое создание таблиц и связей

ИНСТРУМЕНТЫ:
  • Maven - система сборки проекта
  • Lombok - уменьшение boilerplate кода
  • Git - система контроля версий


3. АРХИТЕКТУРА ПРОЕКТА
───────────────────────────────────────────────────────────────────────────────

Проект следует принципам многослойной архитектуры (Layered Architecture):

┌─────────────────────────────────────────────────────────┐
│  ПРЕЗЕНТАЦИОННЫЙ СЛОЙ (Presentation Layer)              │
│  • HTML/CSS/JavaScript - пользовательский интерфейс     │
│  • REST Controllers - обработка HTTP запросов           │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  БИЗНЕС-ЛОГИКА (Service Layer)                          │
│  • ProductService - логика работы с товарами            │
│  • CategoryService - логика работы с категориями        │
│  • AdminService - аутентификация администратора        │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  СЛОЙ ДОСТУПА К ДАННЫМ (Data Access Layer)             │
│  • ProductRepository - работа с таблицей products      │
│  • CategoryRepository - работа с таблицей categories  │
│  • AdminRepository - работа с таблицей admins          │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  БАЗА ДАННЫХ (Database Layer)                          │
│  • H2 Database - хранение данных                       │
└─────────────────────────────────────────────────────────┘


4. СТРУКТУРА БАЗЫ ДАННЫХ
───────────────────────────────────────────────────────────────────────────────

ТАБЛИЦА: categories
  • id (Long, Primary Key, Auto Increment)
  • name (String, Unique, Not Null)

ТАБЛИЦА: products
  • id (Long, Primary Key, Auto Increment)
  • name (String, Not Null)
  • description (String, до 1000 символов)
  • price (Double, Not Null)
  • quantity (Integer, Not Null)
  • category_id (Long, Foreign Key → categories.id)
  • image_path (String) - путь к изображению товара

ТАБЛИЦА: admins
  • id (Long, Primary Key, Auto Increment)
  • username (String, Unique, Not Null)
  • password (String, Not Null)

СВЯЗИ:
  • Product → Category (Many-to-One)
  • Category → Product (One-to-Many)

6. ФУНКЦИОНАЛ АДМИНИСТРАТИВНОЙ ПАНЕЛИ
───────────────────────────────────────────────────────────────────────────────

АУТЕНТИФИКАЦИЯ:

УПРАВЛЕНИЕ КАТЕГОРИЯМИ:
  ✓ Просмотр всех категорий в таблице
  ✓ Создание новой категории
  ✓ Редактирование существующей категории
  ✓ Удаление категории

УПРАВЛЕНИЕ ТОВАРАМИ:
  ✓ Просмотр всех товаров с изображениями
  ✓ Добавление нового товара
  ✓ Редактирование товара (включая изображение)
  ✓ Удаление товара

ЗАГРУЗКА ИЗОБРАЖЕНИЙ:
  • Загрузка через форму
  • Сохранение в папку /uploads
  • Генерация уникальных имен файлов (UUID)
  • Предпросмотр изображения перед сохранением

7. REST API ENDPOINTS
───────────────────────────────────────────────────────────────────────────────

ПУБЛИЧНЫЕ API:
  GET  /api/products              - получить все товары
  GET  /api/products/{id}        - получить товар по ID
  GET  /api/products/category/{id} - получить товары по категории
  GET  /api/categories            - получить все категории
  GET  /api/categories/{id}       - получить категорию по ID

АДМИНИСТРАТИВНЫЕ API:
  POST   /api/admin/login         - вход в админ-панель
  GET    /api/admin/categories    - получить все категории
  POST   /api/admin/categories   - создать категорию
  PUT    /api/admin/categories/{id} - обновить категорию
  DELETE /api/admin/categories/{id} - удалить категорию
  GET    /api/admin/products     - получить все товары
  POST   /api/admin/products     - создать товар
  PUT    /api/admin/products/{id} - обновить товар
  DELETE /api/admin/products/{id} - удалить товар
  POST   /api/admin/upload       - загрузить изображение


8. ОСОБЕННОСТИ РЕАЛИЗАЦИИ
───────────────────────────────────────────────────────────────────────────────

РЕШЕНИЕ ПРОБЛЕМЫ ЦИКЛИЧЕСКОЙ ЗАВИСИМОСТИ:
  • Использование @JsonIgnore и @JsonIgnoreProperties
  • Предотвращение бесконечной рекурсии при сериализации JSON
  • Product → Category (без поля products)

СОХРАНЕНИЕ ДАННЫХ:
  • Файловая база данных H2 (не in-memory)
  • Данные сохраняются между перезапусками

УПРАВЛЕНИЕ ФАЙЛАМИ:
  • Статические ресурсы: /static/
  • Загруженные изображения: /uploads/
  • Конфигурация через WebConfig для доступа к файлам

БЕЗОПАСНОСТЬ:
  • CORS настроен для работы с фронтендом
  • Простая аутентификация для админ-панели
  • Валидация данных на клиенте и сервере

14. ОТВЕТЫ НА ВОЗМОЖНЫЕ ВОПРОСЫ ПРЕПОДАВАТЕЛЯ
───────────────────────────────────────────────────────────────────────────────

В: Почему выбрали H2 вместо PostgreSQL?
О: H2 - встроенная база данных, не требует отдельной установки, 
   идеальна для разработки и демонстрации. Легко переключиться на 
   PostgreSQL для production.

В: Как реализована безопасность?
О: Использована простая аутентификация для админ-панели. В production 
   рекомендуется Spring Security с JWT токенами и хешированием паролей.

В: Почему нет регистрации пользователей?
О: Проект сфокусирован на демонстрации основных возможностей интернет-магазина.
   Регистрация - следующий этап развития проекта.

В: Как обрабатываются ошибки?
О: На фронтенде - try-catch с информативными сообщениями. На бэкенде - 
   стандартные HTTP статусы и обработка исключений.

В: Почему использовали Lombok?
О: Lombok уменьшает boilerplate код (геттеры, сеттеры, конструкторы), 
   делает код чище и читабельнее.

В: Как работает загрузка файлов?
О: Используется MultipartFile из Spring, файлы сохраняются в папку uploads/,
   пути хранятся в базе данных, доступ через статические ресурсы.
